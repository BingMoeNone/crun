参数使用提示 — 提示内容从哪来？

光标停在参数上时，底部信息栏显示额外信息

方案 A：扩展 JSON 字段（需改数据）

在 flags_default.json 中为每个参数新增 usage / tip 字段

{
  "flag": "--model",
  "description": {"zh": "...", "en": "..."},
  "tip": {
    "zh": "支持别名(sonnet/opus/haiku)或完整模型名(claude-sonnet-4-6)。\n默认使用当前设置中的模型。",
    "en": "Supports aliases (sonnet/opus/haiku) or full model names.\nDefault: uses the model from current settings."
  },
  ...
}

优点

  • 信息精确、可维护
  • 多行提示，信息密度高

缺点

  • 71 个参数每条都要写内容
  • 工作量较大
方案 B：自动生成提示（推荐）

根据参数元数据自动合成提示信息

// 对 --model 自动生成:
类型: 单选 | 可选值: sonnet, opus, haiku,
claude-sonnet-4-6, claude-opus-4-7,
claude-opus-4-6, claude-haiku-4-5

// 对 --mcp-config 自动生成:
类型: 文本输入 | 参数: 配置文件路径
示例: /path/to/mcp.json

// 对 --chrome 自动生成:
类型: 开关 | 互斥: --no-chrome

从 Flag 对象已有字段（type, choices, conflicts_with, required_args）合成

优点

  • 零额外数据维护
  • 新参数自动生效

缺点

  • 信息不如手写丰富
  • 排版可能不够优雅
🖥️ 提示在 TUI 中的展示位置
/ 搜索  空格 选中  回车 确认  Esc 退出
──────────────────────────────────────────
── 模型
 ○ --model  当前会话使用的模型 [单选]    ← 光标行
 ○ --effort  设置工作量级别 [单选]
 ○ --fallback-model  默认模型过载时回退 [输入]
── 权限
 ...
──────────────────────────────────────────
类型: 单选 | 值: sonnet, opus, haiku, ... | 当前: (未设置)
  已选 1 项

提示显示在底部状态栏上方，随光标移动实时更新

我推荐 B（自动生成），零维护成本且对新参数即时生效。但如果竞赛需要展示数据丰富度，A+B 混合也可以——JSON 有 tip 就用，没有则自动生成。你怎么看？
